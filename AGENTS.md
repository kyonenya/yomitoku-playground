# スキャンOCRワークフロー メモ

このディレクトリでは、日本語書籍スキャン画像から、YomiToku のOCR位置精度を保ったまま、サイズの小さい検索可能PDFを作るための実験を行った。

## Codex 作業メモ

- このワークスペースでは `shell_command` が `windows sandbox: spawn setup refresh` で失敗することがあるので、すべてのコマンドを `sandbox_permissions="require_escalated"` で実行する。

## 重要な調査結果

- YomiToku の `-f pdf --pdf_quality high` はOCRテキストの位置精度が非常に良い。
- 一方で、YomiToku の `--pdf_quality middle` や `low` はOCRテキスト位置が壊れることがある。PDF生成時は `high` 固定にして、後段でサイズ削減する方針がよい。
- YomiToku の検索可能PDFは、背景ページ画像を 8bit RGB JPEG として埋め込む。そのため、白黒本文だけのページでもPDFサイズが大きくなる。
- ある参考PDF（サイズが小さかった既存PDF）が小さい理由は、背景画像が 1bit 白黒かつ JBIG2 圧縮だったため。
- OCRmyPDF や Ghostscript の一般的な最適化コマンドだけでは、YomiToku のRGB JPEG背景を、OCR座標を保ったまま 1bit JBIG2/CCITT に直接変換するのは難しかった。
- 成功した方法は、YomiToku PDF全体を作り直すのではなく、透明OCRテキスト層を残したまま、背景画像XObjectだけを対応TIFF由来の1bit画像に差し替える方法。
- **OCRフォントの字形除去はサイズに最も効くが、ビューア互換性の問題で今回は採用見送り（2026-06-06 調査）**。背景JBIG2化後、ファイルの約半分はOCRテキスト層に埋め込まれた和文フォント（`MPLUS1p-Medium` 等のサブセット、ページごとに重複）だった（実測: ある20ページ本で 画像455KB / フォント638KB）。比較したABBYY FineReader 14出力が小さいのも、OCR層にフォントを埋め込まず**非シンボリックな実フォント名(`MS-Gothic`/`MS-Mincho`)を参照**しているため（画像はむしろABBYYの方が重い）。検索・コピーに必要なのは `ToUnicode` だけで字形(`FontFile*`)は本来不要。
  - しかし pikepdf の字句操作だけでは「小さい・Acrobat警告なし・テキスト完全」を同時に満たせなかった:
    - **字形(`FontFile`/`FontFile2`/`FontFile3`)だけ削除（名前・フラグは不変）**: テキストは全ビューアで完全（`pdfminer` で元と一致、Acrobatでも同じデコード経路）。サイズ約半分（実測 `1,295,099`→`653,051` バイト, 50.4%。Ghostscript レンダリング差分ゼロ）。**ただし Acrobat が「`AAAAAA+MPLUS1p-Medium` フォントを検出または作成できません」と毎回警告**する（シンボリック`Flags=4`＋埋め込み無しで代替を作れないため）。透明テキストなので表示・印刷・検索の実害は無いが警告がうるさい。
    - **`MS-Gothic` にリネーム＋非シンボリック(`Flags=32`)化**: 警告は消えるが、**Acrobatの検索・コピーで OCR テキストがところどころ欠落**する（2バイトCJK名に単バイト簡易フォントのコードを当てるため誤デコード）。`pdfminer` は `ToUnicode` 優先なので欠落が見えず「成功」に見える罠。**採用不可。**
    - **`Helvetica` にリネーム**: `pdfminer` 時点で日本語が数字/ゴミに化ける（`精神分析へ`→`28928`）。論外。
  - **結論（2026-06-06）**: 字形除去は見送り、フォントは埋め込んだまま残す（`yomitoku.py` は字形除去なしの状態へリセット済み）。将来は `fontTools` で**フォントを埋め込んだまま重複サブセットを統合・最小化**して縮小する案を検討（警告なし・テキスト完全で中間サイズを狙う）。実装時は必ず Acrobat 実機で検索・コピーの欠落有無まで確認すること（`pdfminer` の一致だけで判断しない）。
- **YomiToku `--combine` とフォント埋め込みの比較（2026-06-07 調査）**: `--combine` は、複数ページを持つ1つのTIFF/PDF入力に対してはページ間で埋め込みフォントを共有する。一方、ディレクトリ内の単ページTIFF群を通常実行して、生成された単ページPDFを後から pikepdf で結合すると、Acrobat上では同じフォント名に見えても、内部ではページごとに別のフォントサブセット/`FontFile*` が埋め込まれる。
  - 実測例: `260601_感じられたものの文法` の `out` 直下11枚TIFFを、スクリプト後処理なしのYomiToku素PDFで比較した。
    - 複数ページTIFFを作って `yomitoku -f pdf --pdf_quality high --combine`: `26,696,744` bytes、ユニーク埋め込み`FontFile*` 4個、フォント実体合計 `103,745` bytes。
    - 単ページTIFF群を通常実行し、11個の単ページPDFを後から結合: `26,972,247` bytes、ユニーク埋め込み`FontFile*` 22個、フォント実体合計 `402,111` bytes。
    - 素のhigh PDFではJPEG背景画像が両方式とも約 `26,415,572` bytes と支配的なため、総サイズ差は `275,503` bytes（約1%）に留まる。ただしJBIG2背景差し替え後は画像が小さくなるので、このフォント共有差は相対的に大きく効く。
  - 制約: 現在のワークフローの `out\*.tif` は単ページファイルの集合なので、そのまま `--combine` しても本全体の結合にはならない。`--combine` を活かすなら、事前に複数ページTIFFへ束ねてからYomiTokuへ渡す実験が必要。

## Gotchas：試して失敗した点

- OCR位置を重視する場合、YomiToku の `middle` / `low` は使わない。座標が壊れることがある。
- YomiToku の `--combine` は、1つの入力ファイル内の複数ページを結合するもの。ディレクトリ内の単ページTIFF群を横断して1つにまとめるものではない。
  - 0.13.0 のCLI実装では、ディレクトリ入力時に `path.rglob("*")` で各ファイルを個別に `process_single_file()` へ渡す。`--combine` はその関数内でだけ効くため、結合範囲は「その1ファイルに含まれるページ」まで。
  - `-f pdf --combine` 自体は実装されており、複数ページを持つ1つのPDF/TIFF入力なら1つの検索可能PDFを出せる。しかし現在のワークフローの `out\*.tif` は単ページファイルの集合なので、期待する本全体の結合には使えない。
- 半分解像度CCITT画像を作るとき、`/BlackIs1` を固定値にすると白黒反転することがあった。`img2pdf` が生成した画像の `BlackIs1` を確認して反映すること。
- JBIG2のlossy symbol substitutionは文字化け・文字置換リスクがある。今回のテストでは `jbig2 --pdf` のみを使い、symbol-modeやlossyオプションは使っていない。
- YomiTokuのPDFは、ページMediaBoxが画像ピクセル数のままになる。
  - 例: 600dpiの `3340 x 4432` ピクセルが `3340pt x 4432pt` と扱われ、実寸が約 `1178mm x 1564mm` の巨大ページになる。
  - そのままではA5程度の本でもPDFページサイズが壊れる。
- 定型スクリプトでは、元TIFFのDPIを読み、ページ内容全体を `72 / DPI` で縮小してから、MediaBox/CropBoxを実寸ポイントへ直す。
  - 例: `3340 x 4432` ピクセル、600dpiの場合は `400.8pt x 531.84pt`、約 `141.4mm x 187.6mm` になる。
  - 透明OCRテキスト層と背景画像を同じ変換で包むため、OCR位置関係は保たれる。
- 以前の「MediaBox、コンテンツストリームは触らない」という方針は、OCR座標保護には安全だったが、PDF物理サイズが壊れるため定型スクリプトでは採用しない。
- 日本語コメント入りの `.ps1` は **UTF-8 BOM 付き**で保存する。Windows PowerShell 5.1 は BOM なしを Shift-JIS と誤認してパースエラーになる。
  - ラッパースクリプトを PowerShell にしたのは、`uv` が git bash の PATH に乗らず（winget 配置のため）`.sh` から呼べなかったため。
